## Observed deployment-status problem

The latest audited commits displayed multiple failed Vercel status contexts whose targets identified the account build-rate limit rather than an application compile failure. Affected public repositories included the monorepo, BuildWorld AI, and WeaveStudio.

This creates two distinct problems:

1. unrelated repository changes can fan out into numerous Vercel deployment attempts;
2. GitHub presents red deployment checks that can be misread as broken code or failed validation.

The remediation must therefore reduce deployment creation itself. An ignored build command alone is not treated as sufficient because it can still create deployment records and consume platform capacity.

## Canonical production authority before remediation

- GitHub remains the editable source of truth.
- `AI-Project-Portfolio/main` remains authoritative for the employer monorepo applications.
- `buildworld-ai/main` remains authoritative for BuildWorld AI.
- `weavestudio/master` remains authoritative for WeaveStudio.
- private `quoteforge-local/main` remains authoritative for QuoteForge Local.
- Vercel deployment history remains the rollback record.

## Initial remediation target

The first implementation stage will introduce, without activating production cutover:

1. a validated project manifest;
2. a deterministic affected-project selector;
3. tests for documentation-only, application-only, shared-root, legacy-artifact, and malformed-input cases;
4. a GitHub Actions workflow design that can deploy only the selected projects;
5. documentation for required Vercel and GitHub secrets;
6. a dry-run output that can be inspected before automatic Vercel Git deployment is disabled.

## Rollback position

Until the first approval checkpoint is passed, the existing Vercel Git integrations and canonical deployments remain unchanged. The remediation branch may be closed without changing production behavior.
